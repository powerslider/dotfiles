# Engineering preferences (global defaults)

Cross-project taste and house rules. Install at ~/.claude/nitpickle/preferences.md.
Applies to any repo with no local `.nitpickle/preferences.md`. A repo-local file
overrides and extends these. Keep language-specific taste in the local file.

## Writing style (all generated text: comments, plans, specs, reviews)

- Short and concise. Minimal cognitive load. No overengineering, no overcomplication.
- Professional tone.
- No em dashes and no semicolons anywhere. Use periods, commas, or separate
  sentences instead.

## Code comments

- Keep comments short and concise.
- No package comments anywhere unless explicitly asked.
- Do not state the obvious, especially about how the code works. Comment the
  WHAT (intent, constraint, invariant), not the HOW.
- A comment earns its place only when the why is non-obvious. Well-named
  identifiers carry the rest.

## No attribution

- Never state or imply that any output was generated by NitPickle, an agent, or
  any AI. No banners, no disclaimers, no "generated by", no "AI-assisted" labels.
- This applies to everything: code, comments, commit messages, PR descriptions,
  review comments, plans, specs, and docs.
- Everything reads as ordinary engineering work. Run records stay local and are
  never posted.

## Patch style

- Small patches. Prefer splitting a large change into reviewable steps.
- Minimal dependencies. Adding one requires a stated reason.
- No clever abstractions. No vibe-code rewrites.
- Plans before implementation for non-trivial work.

## Commit messages

- When asked for a commit message, output only the message, formatted exactly as
  below and ready to copy. Do not stage, commit, push, or run any git write
  command. The user runs the commit.
- Use Conventional Commits, strictly (https://www.conventionalcommits.org/en/v1.0.0/):
  `<type>[optional scope][!]: <description>`, an optional body, then the footer.
  Types: feat, fix, build, chore, ci, docs, style, refactor, perf, test.
  Breaking changes use `!` after the type or scope, or a `BREAKING CHANGE:` footer.
- Subject in imperative mood, no trailing period, 72 characters or fewer.
- Body explains the why, not the what. Wrap at 72 columns.
- The footer must contain, in this order, an issue reference and a sign-off:

  ```
  resolves <issue_id>

  Signed-off-by: <first_name> <last_name> (<email>)
  ```

- `<issue_id>` is the tracked issue for the change. Derive it from the branch
  name or the task context. If none is known, ask before committing.
- The sign-off name and email come from git config (`git config user.name`,
  `git config user.email`). Split the configured name into first and last. Do not
  invent them.
- No attribution in commits. No `Co-Authored-By`, no "generated by" lines.

## Review style

- Direct but collaborative tone.
- Prioritize correctness and maintainability over nits.
- Do not comment on formatting already enforced by a formatter or linter.
- Flag premature abstraction. Prefer concrete until a third call site appears.

## Things I always reject

- Giant unreviewable PRs.
- New dependency without justification.
- Behavior change without a test.
- Public API change without a migration note.
